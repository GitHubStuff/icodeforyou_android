## Context

`MyApp` was verified against the current project source: its `startKoin`
block already declares `androidLogger()`, `androidContext(this@MyApp)`, and
`modules(appModules)` — no `MyApp` change is needed. The only wiring left is
the aggregate itself: `appModules` goes from `emptyList()` to a list whose
first entry is `preferencesModule`.

## Design

- `AppModules` stays the single aggregation point (spec §3): future phases
  append `coinsModule`, `catsModule`, `carsModule`, `databaseViewerModule`
  to this same list. `MyApp` never changes again for DI.
- The list form (`listOf(...)`, one module per line) keeps every future
  addition a one-line diff in review.


Then run the app. Two things prove the graph is live:

1. Launch succeeds — Koin now resolves `preferencesModule` at startup, so a
   broken definition would crash on launch, not later.

No visible behavior change — nothing injects `PreferencesRepository` yet.
Phase 1 plumbing is complete with this step: contract, tests, implementation,
