## Context

Phase 3 opens on the shapes locked in the Cat API exploration: one call —
`GET /v1/images/search` with `has_breeds=1` — serves both list and detail,
so no second fetch on tap. The models mirror that session's approved
design with one CatsCarsCoins adaptation: `has_breeds=1` guarantees breed
data, so `Cat.breed` is **non-nullable** — the API contract encoded as a
type instead of a runtime check.

## Design

- **Two types, one file** — a `Cat` is meaningless without its `Breed`;
  splitting them buys nothing.
- **`id` is the remote String id**, The Cat API's key — not a Room rowid.
  The persistence layer decides its own keying when the entities arrive.
- **FTS contract stated at the source:** the searchable text comes from
  `name`, `origin`, `temperament`, `description` — named in the KDoc so
  the FTS4 entity (upcoming) and its tests build to a written list, not
  an assumption.
- Lean per YAGNI: fields the detail screen renders and the search
  indexes, nothing speculative.

```kotlin
// cats/domain/Cat.kt

package com.icodeforyou.catscarscoins.cats.domain

/**
 * One cat from The Cat API (spec Phase 3): an image plus its breed.
 * The fetch uses has_breeds=1, so [breed] is guaranteed present and
 * therefore non-nullable — an API contract encoded as a type.
 *
 * [id] is the remote image id (String — The Cat API's key, not a Room
 * rowid). Pure domain: no Room, no Coil, no wire knowledge.
 */
data class Cat(
    val id: String,
    val imageUrl: String,
    val breed: Breed,
)

/**
 * Breed facts, lean per YAGNI — exactly what the detail screen renders
 * and the search indexes. The FTS4 searchable text (spec: Cats search)
 * is drawn from [name], [origin], [temperament], and [description]; the
 * data layer's FTS entity mirrors those four.
 */
data class Breed(
    val id: String,
    val name: String,
    val origin: String,
    val temperament: String,
    val lifeSpan: String,
    val description: String,
)
```
