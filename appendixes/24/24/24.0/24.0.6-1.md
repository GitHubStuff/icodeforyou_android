## Context

The Kover plugin (0.9.8) is already applied in `app/build.gradle.kts` and
pinned in the version catalog from Phase 0 standup. This step verifies the
coverage pipeline runs end-to-end: instrument, execute unit tests, emit the
HTML report. Coverage percentage is irrelevant at this gate — only wizard
stub tests exist. The gate proves the *toolchain*, not the numbers.

## AGP 9 caveat

Kover on Android normally generates **variant-suffixed** report tasks —
`koverHtmlReportDebug`, `koverXmlReportDebug`, etc. — and the plain
`koverHtmlReport` task covers all variants. Under AGP 9.0.0, Kover had a
known defect (kotlinx-kover issue #785): variant-specific report tasks were
not generated, and the total `koverHtmlReport` task carried no test-task
dependencies — running it alone produced an empty report (all counters 0).
The fix is merged upstream; whether the pinned 0.9.8 contains it depends on
release timing against AGP 9.2.1.

The gate command below therefore runs the debug unit tests explicitly before
the report task. This form is correct on both fixed and unfixed Kover builds
— explicit test execution collects coverage data regardless of whether the
report task wires its own test dependencies.

## Gate

First check which report tasks exist:

```
./gradlew :app:tasks --group verification
```

**If `koverHtmlReportDebug` is listed** (fixed Kover):

```
./gradlew :app:koverHtmlReportDebug
```

Report: `app/build/reports/kover/htmlDebug/index.html`

**If only `koverHtmlReport` is listed** (unfixed Kover under AGP 9):

```
./gradlew :app:testDebugUnitTest :app:koverHtmlReport
```

Report: `app/build/reports/kover/html/index.html`

## Pass criteria

- The report task completes successfully.
- The HTML report opens and lists project classes
  (`MainActivity`, `MyApp`, theme files) with non-empty class counters —
  covered percentages near zero are expected and fine.
- If every counter reads `missed=0 covered=0` with no classes listed, the
  report is empty — rerun using the explicit-test form above.
