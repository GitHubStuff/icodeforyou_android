## Design

- **One binding:** `single<CoinsRepository> { RoomCoinsRepository(get()) }`.
  The `CoinDao` resolves from `databaseModule` (24.2.8) — cross-module
  resolution is Koin-native; features own their bindings, `db/` owns the
  DAOs.
- Same file shape as `preferencesModule`: feature `di/` package, contract
  bound to implementation, consumers never see the persistence types.
- Placeholder-free: the polling engine and `CoinsViewModel` definitions
  arrive in their own steps rather than as commented stubs.

import org.koin.dsl.module

/**
 * Coins feature wiring (spec: Koin runtime DSL). Consumers inject
 * [CoinsRepository] — the domain contract — and never see Room types
 * (DIP). The CoinDao parameter resolves from databaseModule's single.

 * Grows with the phase: the polling engine and CoinsViewModel register

    single<CoinsRepository> {
