## Context

To back the schemaless layout engine, we require an abstraction layer that exposes the low-level SQLite system tables and executes arbitrary raw queries securely outside the view models. This contract allows the UI to read tabular datasets dynamically without leaking explicit table structures or DAO mappings into presentation space.

## Design

1. **System Metadata Introspection.** `getTableNames()` extracts all valid user-defined database table targets from the `sqlite_master` catalog, omitting metadata system entries like Room internal trackers or FTS helper segments.
2. **Dynamic Row Projection.** `queryTable(tableName)` projection converts variable-width rows into unified `DbRow` schemas dynamically by harvesting header mappings straight from the underlying cursor metadata.
3. **Pure Domain Boundaries.** The contract deals strictly with plain structures (`String`, `List`, `DbRow`) entirely disconnected from Room classes or framework packages, remaining compliant with the app blueprint's strict port architecture.

// dbviewer/domain/DbViewerRepository.kt

package com.icodeforyou.catscarscoins.dbviewer.domain

import com.icodeforyou.catscarscoins.dbviewer.data.DbRow
import kotlinx.coroutines.flow.Flow

/**
 * Domain port for extracting schemaless structural data directly from SQLite tables.
 * Decouples presentation blocks from Room's structured compile-time data objects.
 */
interface DbViewerRepository {

    /**
     * Emits the current list of human-readable physical database table names,
     * filtering out framework shadow structures (e.g., room trackers, fts backings).
     */
    fun tableNames(): Flow<List<String>>

    /**
     * Executes a raw snapshot dump of [tableName], returning its column header array
     * paired with a map-backed list of row items.
     */
    suspend fun queryTable(tableName: String): Pair<List<String>, List<DbRow>>
}
